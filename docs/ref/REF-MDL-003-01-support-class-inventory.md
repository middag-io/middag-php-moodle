# REF-MDL-003-01: Support Class Inventory & Anti-Patterns

> Detail supporting [MDL-003](../decisions/MDL-003-support-layer-pattern.md). Reconstructed from the `moodle-local_middag` legacy vault (`ADR-203`, `ref-203-01`).

## Legacy inventory — 40 support classes across 8 groups

| Group                | Support classes                                                                                                                                                           |
|----------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| Data/Persistence     | `db_support`, `cache_support`, `config_support`, `preference_support`                                                                                                     |
| Auth/Authorization   | `auth_support`, `capability_support`, `session_support`, `role_support`                                                                                                   |
| Context/Entities     | `context_support`, `user_support`, `course_support`, `category_support`, `group_support`, `cohort_support`, `enrol_support`, `custom_field_support`, `competency_support` |
| Content/Files        | `file_support`, `grade_support`                                                                                                                                           |

| Runtime/Infra        | `task_support`, `plugin_support`, `request_support`, `version_support`, `time_support`, `lock_support`, `check_support`, `settings_support`                               |
| Bridge               | `di_bridge_support` (MDL-006), `router_bridge_support` (MDL-017)                                                                                                          |

All 40 were `@internal` in the legacy plugin — consumption only via the generated facade mirror in `facade/moodle/*_support`.

## Current state in this adapter (`middag-io/moodle`)

The OSS extraction grew the inventory to **45 `*Support` classes** in `src/Support/`, plus `Support/Moodle` — a static aggregator that returns fresh `*Support` instances rather than a generated per-class facade. The facade-generation mechanism itself (`build:facades`, per-class mirror) is a **consumer-product concern**, not part of this OSS adapter: MIDDAG product facades are generated by dev-tools inside the consumer plugin (`{component}\facade\...`), never added here (see `CLAUDE.md` gotcha 7). Importing `Support/*Support` classes directly from this package is therefore not an anti-pattern in the same sense as the legacy plugin — there is no OSS-side facade layer to bypass; the anti-pattern is scoped to the consumer product's own facade convention.

## Auth/Capability — dual consumption path (unchanged rule)

`AuthSupport`/`CapabilitySupport`-equivalent classes still follow the `*Support` pattern, but preferential consumption in controllers/services/extensions is via DI against `Security\Authentication`/`Security\Authorizer`/`Security\Capability`'s contracts (see `src/Security/Contract/`). The facade/direct-call fallback remains for procedural contexts without DI (`lib.php`, navigation hooks, install/upgrade) — the same tradeoff the legacy ADR documented, still real today.

## Admission rule (unchanged)

A new support class requires 2+ real call sites in the framework consuming the same subsystem. Single-call convenience methods are accepted only when no type transformation is involved (pure testability wrapper).

## Anti-patterns (legacy `ref-203-01` §10, still applicable in spirit)

- Direct import of a `support/`-equivalent class from a consumer extension, bypassing whatever facade convention that consumer has adopted.
- A support class carrying business logic (should be a pure adapter instead).
- Calling a Moodle API (`$DB`, `$PAGE`) outside a `*Support` class.
- Creating a support class with only 1 call site.
- Using the auth/capability facade fallback in a controller instead of DI.
- A support class depending on `domain/`, `service/`, or `kernel/`-equivalent layers (only `shared/`-equivalent code is allowed).
